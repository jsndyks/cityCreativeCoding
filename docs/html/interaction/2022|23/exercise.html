Core Tasks

OK, we are going to try four tasks that will build on the concepts and code that we introduced in the Interaction Interactive Lecture. Make sure you have understood and completed all of the tasks in there before you try these!

1. Move

The whole idea of studying Interaction is that it enables you (or others) to control a sketch with your mouse and keyboard once it is running.
We can control colours, positions, behaviours ... pretty much anything really.
And that's what many sophisticated sketches do.

Let's start by making something move.

Move Task

Create a new sketch that draws a square in the centre of a square window sized 400 by 400.
Use two global variables float x and float y to specify the position of the square - at width/2, height/2: 

void draw()
{
  rect(x, y, 60, 60);
}

Remember you have to declare those variables at the top of your sketch and you should initialise them with appropriate values inside setup(), so that your square is positioned correctly when your sketch starts.

Run your sketch to check it behaves as expected.

Now add a keyPressed() method to the sketch that maps the w and s keys to commands that increment and decrement the variable y by a value of 1 respectively.
The condition in keyPressed() that gives you the keyboard interaction will look something like this: 

if (key == 'w') {
// -- do something in here that moves the rectangle up!
}
Really think about the coordinate system and how pressing w will move the square up here.

Add conditions to keyPressed() that map the a and d keys to commands that increment and decrement the variable x by a value of 1 respectively.

Run your sketch to check you can move the square around the canvas using the a, d, s and w keys.

This is pretty cool - you have an interactive sketch!
Show your friends, show your tutor.
Celebrate!

But not too much ... because there are a few more features and tricks that you should try later in the challenges section - they will serve you well.
So carry on with these CORE Tasks for now and we can try the Move challenges later.

2. Colour

This kind of interaction can be used to control almost anything in your sketch: shapes, positions, colours, sizes ... all sorts of behaviours.
Let's change the colour of a shape according to input from the keyboard for starters.

Colour Task

Start a new sketch with a void setup() method that sets the size of the sketch canvas to 400 by 400.

Initialise a new variable called red with a value of 128.

Add a void draw() method that contains the following code: 

void draw()
{
  background(100);
  fill(red, 128, 128);
  rect(100, 100, 60, 60);
}

Use Processing's void keyPressed() method to increment the value of red by 10 for every press of the R key.
Add another interaction that decrements the value or red by 10 for every press of the r key.

That's pretty good, but with a few more commands and a little more creative thinking you can create a full RGB colour selector!

You can try this in the Red Green Blue Selector challenge later - but for now ... let's move on to Task 3.

3. Splash

This is a really nice one to try - it's not too difficult, and is pretty open ended. There's lots of room for creative ideas.

Let's use what we have learned to add some interactions that allow you to rapidly change a dynamic display ... and perhaps even get a little creative!



Splash Task

Make a big canvas in setup() and give it a white background().

Add a draw() method that draws a circle with your favourite fill() wherever the mouse is positioned.

The circle should have no outline - noStroke()

A new circle should be drawn wherever the mouse moves, and so your mouse will leave a trail. You must make sure you have no background() command in draw(). This would wipe the background clear in every draw() loop. We want a trail of coloured circles.

Now update the colour of the circle depending on the position of the mouse - so that there is more red in the colour as the mouse moves right and more blue in the colour as the mouse moves down.
Just use mouseX and mouseY somewhere in the RGB colour definition.

This should look quite cool and colourful - particularly if you have a circles with some transparency. Remember, the fourth argument given to the fill() command is the alpha channel, which sets the transparency.

Nice. You could make a spray paint app out of this. The Colourier Spray Splash challenge asks you to do just that, and gives you some ideas as to how you might go about doing so.
Try it later, after you have completed the four CORE tasks - nearly there.
But for now ... we're hunting ...

4. Target

We can use the detection of events, and the creation of shapes on screen to produce simple games - like a target hunter.

One very useful method is dist() - which calculates the distance between two points.
The Distance2D sketch shows how nicely this can be tied to interactions.
Try it - just follow 👆 that link and interact.

But let's not get ahead of ourselves!
For now, rather handily, dist() allows us to calculate the distance between the mouse position and the centre of a circle. This can tell us whether the mouse is either:

  • inside the circle - closer to the circle's centre than its radius, or
  • outside the circle - further from the circle's centre than its radius

A key code-snippet looks something like : 

// -- check on distance from mouse to circle centre:
if (dist(circleX,circleY,mouseX,mouseY) < radius) {
  // -- within circle - do something
  fill(255,128,64);
} else {
  // -- outside circle - do something else
  fill(255);
}

// -- draw the circle
ellipse(circleX,circleY,radius,radius);
Target Task

Write code that produces a circle in your favourite fill() colour at a random position on the screen.
Each time you move the mouse inside the circle highlight it in some way, with a different fill() or stroke() or strokeWeight().

When you click inside the circle, redraw it at a random position within the sketch and at a random size.

Before you start writing code consider the global variables and the functions you will need to implement the behaviour of your sketch. A good approach would be break the behaviour of the sketch down into smaller testable goals, for example:

  • Make a list of global variables and discuss it with one of the lab staff;
  • Draw a centred circle and test that it's in the right place - draw it at (0,0) or add a point() in a different colour at the centre after you have drawn it);
  • Add the click detection and test - based on the code-snippet provided above;
  • Update the circle position;
  • Update the circle size;
Hints:
  • To detect whether the click is within the circle you can use Processing's dist() method.
  • To test whether the mouse is in the circle consider printing "inside" to the console when it is.
  • To test for the click detection consider printing "Click" to the console when you click the mouse button.
  • To test whether the click is inside the circle consider printing "Hit" or "Miss" to the console.
  • To set the position of the circle to a random position you can use Processing's random() method.
  • Make sure you constrain the positions and the circle size so the circle stays within the sketch and is big enough to see and click!

Super Move Challenge

If you want to push things a bit, and learn a little more, by enhancing this interactive sketch then try this challenge that will extend the sketch to constrain!, report! and change!:

Constrain!

It's pretty easy to keep the shape on the canvas ...

Extension 1a: Add if statements to prevent the values of x and y becoming less than 0 or more than 400. Use width and height (rather than 400) so that things still work if the sketch changes size!

Extension 1b: Add if statements to allow the square to go up to, but not past, the bounds of the sketch.

Extension 1c: Can you find and use a Processing command that will constrain the extents of x and y in a single method?
This may work better, and be clearer to read, than all of those if statements.

Report!

We probably want to know something about what is going on in the sketch ...

Extension 2a: Add code to the draw() method that displays the x and y position of the rectangle in the bottom corner of the sketch.
The text should update when the position of the rectangle is changed.

Extension 2b: Add conditions to the keyPressed() method that move the square more quickly if CAPITAL A, D, S or W are pressed.
Just add four more conditions based on the value of key.

Extension 2c: Add code to the draw() method that displays the letter of the most recently pressed key inside the rectangle.

Change!

Add an interaction that changes the status of the sketch, and the behaviour of the sketch as a result ...

Extension 3a: Can you change the colour of the shape as it moves? Use fill() and vary the parameters according to the position of the shape.

Extension 3b: Add a boolean global variable called isSquare with isSquare = true;
This will store information about the status of your sketch: is the shape a square or not?
Well, at the moment it is, but can you add an interaction that enables us to vary this?
For example, use a key interaction to toggle isSquare from true to false.


Extension 3c: Add a condition to the keyPressed() method that changes isSquare to false if you click a particular key.

Extension 3d: Now add a condition in the draw() loop that checks whether isSquare is true and draws a rect() (a square one) or an ellipse() dependent upon this.

Extension 3e: You have probably figured out by now that you'll need another condition in keyPressed() that enables you to make isSquare true again - so add one of these!
Note that you can toggle a boolean variable by setting it to its inverse: isSquare = !isSquare;

Extension 3f: So how would you add an option to make the shape a triangle()?



2+. Colour Selector

Can you extend the colour specification functionality a little to produce something much more useful.

Colour Selector Challenge

Extension 1: Add statements that prevent the value of red becoming less than 0 or more than 255.

Extension 2: Add statements that change the green and blue colour components with G/g and B/b.

Extension 3: Add statements that show the currently selected colour by adding text() that contains a String that shows all three colours - e.g.

// -- initialise a string to hold colour details
String colourDefinition = "";

// -- update with the current colour:
colourDefinition = "rgb("+red+","+green+","+blue")"
// -- the + signs above just stick the text together
// -- so "Lon"+"don" = "London"

// -- show the current colour on the canvas:
text(colourDefinition,width/2,2*height/3);"

3+. Spray Splash

You have the basics, but we can do loads more with this sketch with a few more commands!

Spray Splash Challenge

Enhance the sketch by varying the colour depending upon whether the r, g or b keys were most recently pressed.

  • r - red varies with x, green varies with y
  • g - green varies with x, blue varies with y
  • b - blue varies with x, red varies with y
This will involve a condition that depends upon what is held in key and you'll need if and else if to control what happens dependent upon this interaction.

Have a play and then decide whether you want to really go for it by ...
Varying the colour in three additonal ways depending upon whether the R, G or B keys were most recently pressed.

  • R - red varies with x, blue varies with y
  • G - green varies with x, red varies with y
  • B - blue varies with x, green varies with y
This will involve extending the condition that depends upon what is held in key.

Then think about some other things you might do here ...

  1. Vary the opacity of the colour depending on a mouse click - this is the fourth argument given to a fill() command - the alpha channel.
  2. Change the size of the circle depending upon some mouse or keyboard interaction - you may want to constrain() this to a sensible range!
  3. Add a random() element that positions the circle around the mouse position rather than directly on it.
    If you get this right, it should emulate a spray can.
    Of course, the amount of randomness itself could be determined by a interaction through mouse or keyboard.
  4. Add some other randomness!
There really are lots of possibilities, even given the limited commands we have explored already.
Be creative!
