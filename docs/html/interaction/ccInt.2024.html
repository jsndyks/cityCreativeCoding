2. Keyboard Input

Happily, Processing sketches are aware of what's going on with the keyboard. You can find this out and use this in your sketches in a couple of ways. Firstly, Processing retains a global variable (like mouseX and mouseY) called key. This is a character variable that holds the value of the most recently pressed key on the keyboard. It updates whenever a key on the keyboard is pressed - you don't have to ask it to do so, Processing just takes care of it while any sketch runs. It has been there the whole time! Useful!

#3 - keyPressed() : a method that runs when a key is pressed

One of the most useful functions for interaction is the keyPressed() method.

This method is called and run by a Processing sketch every time a key is pressed.

This is important so let's say it again: any method that you specify in a sketch, that you give the name keyPressed(), will be run by your sketch every time you press any key on the keyboard.

This is so important that I am going to say it again in bold: any method that you specify in a sketch, that you give the name keyPressed(), will be run by your sketch every time you press any key on the keyboard.

You simply create a new method called keyPressed(), just as you have made methods to do all sorts of other things and as shown below, and Processing will automatically call this method every time it detects a key press interaction: every time a key is pressed on the keyboard
Let's try it!

    // -- set up the things that don't change ...
    void setup() {
    size(400, 400);
    background(220);
    textSize(18);
    fill(64);
    }

    // -- nothing in here for now - an empty draw loop ...
    void draw() {
    // -- this still runs 60 time per second!
    }

    // -- Processing calls this method whenever a key is pressed:
    void keyPressed() {
    // -- put the value of 'key' in a random position:
    text(key,random(20,width-20),random(20,height-20));
    }

Ordered Characters Sketch

This is pretty, but we might want to do something a bit more structured with the input, like arrange it in a way that shows sequence.

To do this we'll need to store information about where we put each piece of text() that we generate in response to a key press being detected in keyPressed().
Hopefully you'll be thinking global variables as you are reading this?

We'll use a global variable called p to store this positional information about how far along the canvas we want to plot our first letter and increment it by a few pixels each time a key is pressed - in the keyPressed() method so that subsequent letters are printed a bit further along.

This should move each successive letter to the right. That is fine while p is less than width. But our letters will plot off the right hand side of the canvas as we type more keys unless we do something about this!

A little bit of simple maths will work out where we put each successive value of key to the right of the next ... until we get to the end of the screen canvas, at which point we need to head back to the left and down a line.

It's pretty straightforward ...

  • p % width - uses the modulo operator %.
    This calculates the remainder when one number is divided by another.
    So 110%100 is 10.
    As is 210%100.
    This is handy as any number %width gives us a position between 0 and width. When we get a number that is just more than width, this gives us a position back on the left hand side of the canvas as (width+1)%width is 1>!
  • 24*int(p/width) - firstly divides p by width. It also uses the int() operator to convert the result to an integer. This means that we get a whole number - the number of times width completely fits into p.
    So int((width+1)/width) is 1!
    And int((5*width+1)/width) is 5.
    It follows that int(110/100) is 1. And int(210/100) is 2. If you think about it, this means we find out how may times p has filled our width through this part of the calculation. Which tells us how many lines of length width the characters have taken up. If we then multiply through by a number that is bigger than our current letter height, like 24 or something, this will position text that has a p value that is greater than width down the screen so that it does not overlap. I have added another 100 on in the script that follows so that the text does not start at the very top of the screen.

Here's some code that should do the job!

    // -- global variable - used to count keypresses and position text
    int p = 0;

    void setup() {
    size(400, 400);
    background(220);
    textSize(18);
    fill(64);
    }

    // -- do nothing in the draw loop for now!
    void draw() {}

    void keyPressed() {
    p+=12; // -- add 12 to p each time

    // -- put the char held in 'key' at a screen position
    // --   that increases in x and y with each key press
    text(key,p%width, 100 + 24*int(p/width));

    // -- the above just calculates a position for the text as p increases
    // -- use println() to print out the values to the console if you like
    // -- remove '/width' and see what happens as you get to the right hand side!
    }

KeyPressed Sketch with Control Flow

This is a good start, but we are likely want to control program flow from the keyboard.

  void keyPressed() {
    // -- change program flow depending on value of key:

3. What About the Mouse?

So, we have a useful series of variables and methods to enable us to interact with and through the keyboard. But that only gets us so far - we can only specify a limited set of things as defined by the keys.
What if we want to identify locations on the screen.
Quickly.
With precision?

Well, in the 1970s researchers at the Xerox Research Centre in Palo Alto, California were asking themselves the same question:

What if we want to identify locations on the screen?
Quickly. With precision?
How do we do that?

So they invented the mouse!

It's hard to imagine just how innovative this mode of interaction was back then, and the kind of computing and interaction it has enabled.

One of these innovators was Stuart K. Card, who worked in and subsequently led a hugely creative and productive team at Xerox PARC from the mid 1970s until 2010. One of the many things that he and his colleagues were thinking about back then (probably more deeply than we just did) was how to get computers to know what we need of them. They wanted to come up with devices that were easy to use and provided efficient and precise interactions.

In their mouse prototypes, wheels and rollers generated electronic signals (and squeaked a bit sometimes), that reliably allowed a computer user to specify positions on a desk, that could in turn be converted into positions on a screen. This gives us an effective flexible and efficient new means of interacting with computers.

#5 - Summary : mouse (cursor) variables & methods in Processing

These methods and variables are in some ways similar to those used with the keyboard. And you know some of them and have already used them in some of your sketches. Let's start with a summary table.

Event handling : mouse

Variables mouseX A global float variable that holds the x coordinate of the mouse pointer.
mouseY A global float variable that holds the y coordinate of the mouse pointer.
mousePressed A global Boolean variable that is true while the mouse left button is pressed.
mouseButton A global variable that is equal the constant LEFT when the left button is pressed and RIGHT when the right button is pressed.
Methods mouseClicked() A method that is called in response to the mouse left button being pressed and released.
mousePressed() A method that is called in response to the mouse left button being pressed.
mouseReleased() A method that is called in response to the mouse left button being released.
mouseMoved() A method that is called in response to the mouse being moved with no buttons being pressed.
mouseDragged() A method that is called in response to the mouse being moved while the left button is pressed.

OK, so you already know the global variables mouseX and mouseY. These tell you something about how your sketch is being interacted with.

Then we get to the methods, which are called whenever we click, press, release, move or drag the mouse. Or whatever peripheral we are using to control where on the screen we want to focus.

Detecting, Recording and Doing - How to Structure a Sketch for Interaction

Often people don't read those big red titles üëÜ.
So I am going to write it again in black here.
This section is about: "How to Structure a Sketch for Interaction".
And you need to know this in order to make reliable interactive sketches.

The key thing is that empty draw() loop!

The draw() loop is the place to draw and do things.
The event handling methods like keyPressed() and mouseClicked() are not the place for this.
Do not draw or do things in event handling methods like keyPressed() and mouseClicked()
 These methods are best for changing and recording system state - the way the sketch is configured at any time.
The draw() loop is then the place to act on that state - to do the things requires by the interaction.

In summary ... (actually read through this code üëá) ...

// -- GLOBAL VARIABLES
// -- Declare and initialise variables that describe
// --   the initial state of your system, here.
// -- You will vary these to record the way that the state
// --   of your system varies over time in response to interactions.

// -- THE MAIN DRAW LOOP!

void draw() {
  // -- Do things in here
  // --   based upon variables set in the event handling methods
  // --   that describe the system state.
}

// -- THE EVENT HANDLING METHODS!

void keyPressed() {
  // -- Set variables here that describe the state of your sketch
  // --   based upon interactions and input.
  // -- These will be acted on in the draw() loop.
}

void mouseClicked() {
  // -- Set variables here that describe the state of your sketch
  // --   based upon interactions and input.
  // -- These will be acted on in the draw() loop.
}

So ... even more bluntly ... Don't draw things in the event handling methods!.

Here is a refactoring of the keyPressed sketches that structures things as we advise above.

It's a little longer than the other examples, but should be readable.
Take your time to read it through and see how we have refactored things in line with this new advice.

// -- GLOBALS : declare and initialise global variables
// --   that describe system state - false before interaction:
boolean clearScreen = false;  // -- do we clear the screen?
boolean addText = false;      // -- do we add text?
boolean isRed = false;        // -- do we make text red?

// -- SET UP : initialise and do things that don't change :
void setup() {
  size(400, 400);
  background(220);
  textSize(18);
}

// -- ACT : do things in here (like draw) based upon system state!
void draw() {

  // -- check whether to make text red:
  if (isRed) {
    // -- set fill to red:
    fill(255, 0, 0);
    isRed = false;
  } else {
    fill(64);
  }

  // -- check whether to add text:
  if (addText) {
    // -- add text at the mouse position:
    text(key, mouseX, mouseY);
    addText = false;
  }

  // -- check whether to clear screen:
  if (clearScreen) {
    // -- clear the screen: random light grey!
    background(int(random(180, 240)));
    // -- reset system state:
    clearScreen = false;
  }
}

// -- INTERACT : detect things in these event handling methods
// --   that establish and record system state:
void keyPressed() {
  // -- set variable to record current state:
  addText = true;

  // -- set variable to record current state:
  if (key == 'r' || key =='R') {
    isRed = true;
  }
}

void mouseClicked() {
  // -- set variable to record current state:
  clearScreen = true;
}

One of the reasons we didn't organise the other examples in this way was to keep them short and sweet as we introduced concepts.

  • use it to set variables that describe the state of your system (sketch)
  • check these in the draw() loop to control flow based upon inputs acquired through interaction

#9 - HUH? : that's not a graphical user interface?

We have deliberately not focussed on some of the standard windows style interface elements here - we're getting under the bonnet of these interfaces to give you scope for making more interesting, innovative and inspiring interactions!
But maybe we should show you how you can use what you know to make them!

Here's a basic example:

int x, y, size;
boolean isIn = false;    // -- is the mouse in the rect()?
boolean isOn = false;    // -- is the checkbutton on?

void setup()
{
  size(400, 400);
  x = width/4;
  y = height/4;
  size = width/15;
}

// -- The draw() loop - here is where we control flow and draw things!
void draw()
{
  if (isOn) {
    // -- yellow if on!
    background(255, 255, 140);
  } else {
    // -- white if not!
    background(255);
  }

  // -- change colour to be used in rect based on status:
  if (isIn) {
    fill(255, 128, 64);
  } else if (isOn) {
    fill(255, 0, 0);
  } else {
    fill(240);
  }

  // -- the main rectangle - always shown :
  rect(x, y, size, size);

  fill(64, 200);
  // -- things to show based on status!
  if (isOn) {
    // -- middle rectangle and reminder text :
    rect(x+size*0.25, y+size*0.25, size*0.5, size*0.5);
    text("Remember to turn me off before you leave ...", 120, height-10);
    text("On", x+size*1.5, 15+height/4);
  } else {
    text("Off", x+size*1.5, 15+height/4);
  }
}


// -- Event handling methods:
// --   here is where we detect interactions and capture them in variables...

// -- Processing will call this method whenever the mouse is moved:
void mouseMoved()
{
  // -- is mouse position inside the rectangle?
  if  ((mouseX > x/2) && (mouseY >y) && (mouseX < x+size) && (mouseY < y+size))
  {
    isIn = true;
  } else {
    isIn = false;
  }
}

// -- Processing will call this method whenever the mouse is clicked:
void mouseClicked()
{
  // -- is mouse position inside the rectangle?
  if  ((mouseX > x/2) && (mouseY >y) && (mouseX < x+size) && (mouseY < y+size))
  {
    // -- switch isON to whatever it is not - shorthand follows on the next line (try it):
    // -- isOn = !isOn
    if (isOn) {
      isOn = false;
    } else {
      isOn = true;
    }
  }
}

The key thing to see here is the way that we check whether the mouse is in the rectangle or not with some basic comparisons of the mouse position with the extent of the rectangle.
