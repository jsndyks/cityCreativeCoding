This section introduces the if statement: a programming language construct which allows you to write code that makes choices as it executes.

By the end of this session you should be able to:

  • write if statements in your code so that it makes decisions, depending upon conditions, as it executes;
  • use if statements to choose between different actions based on the mouse position;
  • write nested if statements and else-if statements to make decisions based upon complex and composite conditions;
  • use a variety of different kinds of boolean expressions in the conditions of your if statements;
  • use randomness and if statements to simulate throwing dice or tossing coins.

1. Coding programs that adapt to context

We may face two situations when writing code:

  1. We may know everything that is likely to happen when code is running, everything is entirely predictable.
    We know all data values and how they should be processed, everything is predetermined.
    This would make coding pretty straightforward, but it doesn't happen very often.
  2. There are things we don't know when we write the code, that will only transpire as the code is running, but they are "known unknowns".
    We know what kind of data our code has to process and we know the rules for how it should be processed when it becomes available.
    This is usually the case.

As the second of these situations is the one we normally have to deal with as programmers, we are expected to write code that is flexible enough to process many alternative situations - combinations of inputs and user behaviours by applying known rules. The precise context only becomes known as the code executes, but the rules can be figured out in advance.

Variables and parameterization is one way to achieve this. You have already drawn code that behaves differently when the user changes their input (e.g. we drew rectangles that traced the user's cursor). However, parameterization can only take you so far because incoming values are processed in a uniform way.

A fundamental programming concept that brings us much closer to fully flexible programs is the the if statement

2. Example: choose a shape based on mouse position

void setup() 
{
  size(400, 300);
  fill(203, 118, 122);
}

void draw()
{
  background(255, 236, 149);

  if (mouseX < width/2)
  {
    ellipse(mouseX, mouseY, 80, 80);
  }
  else
  {
    rect(mouseX, mouseY, 80, 80);
  }
}

Execution of an if statement happens in two stages:

  1. First the condition is evaluated. The condition is the stuff inside the round brackets, immediately after the keyword if.
    In this example the condition is mouseX < width/2.
    A condition can be any boolean expression (more about these in a little while)
  2. Then ...
    • Either the first block of code is executed (the code in between the condition and the keyword else),
    • Or the second block of code is executed (the code immediately after the keyword else).
    The first block of code will be executed if the condition evaluated to true, the second if the condition evaluated to false.

Some terminology: the first block of code is known as the true branch or if branch and the second is known as the false branch or else branch.

4. Nested Ifs

The two branches of an if statement contain instructions. So far we have become used to quite short sets of instructions that we can write on a single line, such as rect(100, 200, 50, 50); or fill(255);. However, the if statement is an instruction, just one that we typically write on multiple lines. So, there is no reason why one of the instructions contained inside our branches cannot be an if statement itself. 

void setup() 
{
  size(400, 300);
}

void draw()
{
  background(255);

  if (mouseX < width/2)
  {
    if (mouseY < height/2)
    {
        fill(255,0,0);
    }
    else
    {
        fill(0,0,255);
    }
    rect(mouseX, mouseY, 80, 80);
  }
  else
  {
    ellipse(mouseX, mouseY, 80, 80);
  }
}

5. Boolean expressions

Just like an arithmetic expression (such as 2 + 7 or y * (z - 3)), a Boolean expression is evaluated to find its value. But there are only two possible values for boolean expressions: true and false. The simplest Boolean expressions of all are just the Boolean literals themselves, true and false (though we don't often use them in code). More useful are the comparison expressions:

e1 < e2
 evaluates to true if the value of e1 is strictly less than that of e2, else evaluates to false
e1 <= e2
 evaluates to true if the value of e1 is less than or equal to that of e2, else evaluates to false
e1 > e2
 evaluates to true if the value of e1 is strictly greater than that of e2, else evaluates to false
e1 >= e2
 evaluates to true if the value of e1 is greater than or equal to that of e2, else evaluates to false
e1 == e2
 evaluates to true if the values of e1 and e2 are equal, else evaluates to false if they are different
e1 != e2
 evaluates to true if the values of e1 and e2 are not equal, else evaluates to false if they are equal

  3. (2*32) + 17 >= 64 + 17

And we can build more complex Boolean expressions using the Boolean operators to combine simple ones.
In the following table, b, b1, b2, etc. stand for any Boolean expression:

b1 && b2
 Boolean AND: evaluates to true if both b1 and b2 evaluate to true, else evaluates to false
(Note: evaluation proceeds left to right; b2 never gets evaluated at all if b1 evaluates to false; sometimes this matters.)
b1 || b2
 Boolean OR: evaluates to true if either b1 or b2 evaluate to true, else evaluates to false
(Note: evaluation proceeds left to right; b2 never gets evaluated at all if b1 evaluates to true; sometimes this matters.)
!b
 Boolean negation (NOT): evaluates to true if b evaluates to false, else evaluates to false if b evaluates to true

6. Exploring randomness: coins and dice

Suppose we want to write a sketch called HeadsOrTails that simulates the effect of tossing a coin. We need some source of randomness, so that our code will sometimes print HEADS, sometimes TAILS, with both outcomes equally likely, but we can't predict which one we will get on any particular run of the sketch. Here's our first attempt: 

float randomNumber;

void setup()
{
  size(200, 200);
  fill(0);
  textSize(48);
  textAlign(CENTER);

  // Initialise randomNumber between 0 and 2
  randomNumber = random(2);
}

void draw()
{
  background(230, 255, 230);

  String message;

  message = "TAILS";

  text(message, width/2, height/2);
}

The code above declares a variable randomNumber and initialises it in setup(). The command random(2) returns a random number in the range 0 to 2. We can't predict what number we will get (that's the whole point!) but we do know that we are equally likely to get a number between 0 and 1 as we are to get a number between 1 and 2 (and if we do get one between 0 and 1, it is just as likely to be between 0 and 0.5 as it is between 0.5 and 1, and so on; this is known as a uniform distribution).

Heads or Tails?

The draw() code doesn't actually use the random number yet - it just prints the message TAILS every time.

Modify the code so that the message is TAILS if the number is less than 1 and HEADS otherwise.

Now let's suppose we want six randomly chosen possible outcomes, rather than just two. In other words, suppose we want a six-sided die, rather than a coin.

First, it's helpful to know that we can expand the range of our random number so that we get a number in the range 0.0 to 6.0 (see the code below how this can be done). Then we can map each unit interval in our range to one outcome of our roll.

In practice we can achieve this mapping by writing mutually exclusive Boolean expressions that test whether our number is in each of the six intervals in turn: 

float ran;

void setup()
{
  size(200, 200);
  fill(0);
  textSize(48);
  textAlign(CENTER);

  ran = random(6);
}

void draw()
{
  background(255);
  String message="";

  if (ran >= 0 && ran < 1)
  {
    message = "ONE";
  }

  if (ran >= 1 && ran < 2)
  {
    message = "TWO";
  }

  if (ran >= 2 && ran < 3)
  {
    message = "THREE";
  }

  // Can you write the code for the remaining cases?

  text(message, width/2, height/2);
}

7. Choosing from multiple options: else if

When we have more than two options to choose from, as was the case with our previous example, it's common to use the else if. This gives a way of writing multiple, mutually exclusive branches within the same if statement.
The following code snippet implements the roll of the dice using the else if variation: 

if (ran < 1)
{
  message = "ONE";
}
else if (ran < 2)
{
  message = "TWO";
}
else if (ran < 3)
{
  message = "THREE";
}
else if (ran < 4)
{
  message = "FOUR";
}
else if (ran < 5)
{
  message = "FIVE";
}
else
{
  message = "SIX";
}

Note that despite there being multiple alternative branches, it's still the case that only one branch can execute. This is the case even though in some cases multiple conditions may be true (as is the case in the example above). In an else if the successive conditions are tested in order, one at a time. The branch of the first one to evaluate to true is executed, after which the entire if stops. Thus, even if any subsequent conditions are true, they are never evaluated and their branches never executed.

What if?

We just learned that the branch of the first statement in an IF, ELSE IF to evaluate to true is executed, after which the entire if stops. So only one branch is ever executed. If any subsequent conditions are true, they are never evaluated and their branches never executed.

But what would happen in the above program if this was not the case - if any branch that was true in an ELSE IF evaluated?

8. Layout

An alternative and more compact way of arranging the curly braces is the following: 

if (ran < 1) {
  message = "ONE";
} else if (ran < 2) {
  message = "TWO";
} else if (ran < 3) {
  message = "THREE";
} else if (ran < 4) {
  message = "FOUR";
} else if (ran < 5) {
  message = "FIVE";
} else {
  message = "SIX";
}

One more note is that when there is just one statement in a branch, the curly brackets are optional. In fact, since in the code above all branches contain single statements, we could remove all of them! But ...

DON'T DO IT!
ALWAYS WRITE THE CURLY BRACKETS, even when they are optional

Because if you ever decide to add more statements to a branch it is very easy to forget to put the curly brackets in, and this can lead to some very frustrating bugs in your code that are very hard to see!

9. Tips

Debugging: As your sketches become more ambitious you will find that they are harder to get right first time. The art of identifying the errors ("bugs") (🪲) in your code, and fixing them, is known as debugging. Bugs can be frustratingly hard to track down. A common experience for programmers is that we find ourselves thinking:

"But it makes no sense that my code does X !. I can see that it must do Y."

It is tempting to think that the problem must lie elsewhere, that the bug is not in our code but in Processing, or Windows, or the computer hardware. But, in fact, it always turns out that the bug really is in our code. (OK, not always. Very, very occasionally the blame actually does lie elsewhere.)

Here are three techniques that can help:

  1. Talk through your code with a fellow programmer.
    Explain how you expect your code to behave, why you think it should behave that way, and what is actually happening. It is surprising how often the simple act of trying to explain your code to another programmer allows you to see why it is wrong (even when the other programmer can't see the problem).
  2. Reveal the values of variables and to give a visible indication of control flow.
    For example, suppose you have variables x and y, which are used to position the drawing of your widget, but for some unfathomable reason your widget does not pop up as you expect it to when the score goes above 999. You might add use println() to reveal the values: 
    println("value of x:", x);
println("value of y:", y);
    just before the code which draws the widget.
    • If you see the message "value of x: 2000" and your window is 600 pixels wide, then you are on your way to locating the bug.
    • On the other hand, if you don't see any message, then you know that your widget drawing code is not being executed at all. Perhaps that Boolean expression which checks the score is wrong?
    You can also use text() to display the values of variables on the Processing canvas. Sometimes we create a whole list of values that change dynamically as we are debugging. 
    if (inDebugMode) {
  text("x is "+x, 20, 20);
  text("y is "+y, 20, 40);
  text("frameCount is "+frameCount, 20, 60);
  3. Learn to use the Processing Debugger tool as part of your programming workflow.

Make your boolean expressions legible. You can do this best in two ways. First, split long and complicated boolean expressions into multiple simpler ones, store the results of those in boolean variables, then combine the variables back into the larger expression.
Second, use variable names that make your condition read naturally in plain English. Here's an example of turning a rather lengthy condition into one that reads well: 

// if ((mouseX < width/2 && mouseY < height/2) || (mouseX > width/2 && mouseY > height/2))
// {
//   println("upper left or lower right quadrants");
// }
//
// can be rewritten as

boolean mouseUp = mouseY < height/2;
boolean mouseRight = mouseX < width/2;
boolean mouseUpperLeft = mouseUp && mouseLeft;
boolean mouseLowerRight = !mouseUp && !mouseLeft;

if (mouseUpperLeft || mouseLowerRight)
{
  println("upper left or lower right quadrants");
}

Another benefit of using variables to break up complex conditions is that they become easier to debug.

Don't put semicolons in the middle of your if statement It's quite common for novice programmers to develop a 'semicolon reflex' and place one after the if (condition) line. Resist it and remember that the if statement is a single command that we tend to write on multiple lines. Don't through a semicolon in the middle of it.

Bouncing Ball

In this exercise and the next you will develop an animated ball bouncing inside a box.
We start by adapting our scrolling text example : 

float xPosition;  // -- Horizontal starting position on 'x' axis in coordinate space
float xVelocity; // -- Horizontal speed in x-direction

void setup()
{
  size(400,300);
  fill(255,177,8);
  textSize(48);

  xPosition = width/2;  // -- Initialise xPosition to centre of sketch
  xVelocity = -2; // -- Initialise speed in x-direction to -2 (moving left)
}

void draw()
{
  background(64);

  String message = "The next station is Angel...";

  xPosition = xPosition + xVelocity;  // -- Change position in x on each redraw by velocity in x

  text(message,xPosition,(height/2) + 20);
}

    Use a radius of about 50 pixels (you can change this later but, for now, a nice big fat ball makes it easier to see what is happening).
    Best practice is to add an extra variable radius to your sketch (initialise to 50 in setup()) and use that in draw() instead of the literal 50.

11. Summary

  • if statements are used in code that makes decisions as it executes.
  • A single if statement chooses between just two alternative branches of code based on the value of a boolean expression.
  • Boolean expressions evaluate to either true or false (there is no middle way).
  • By nesting and chaining multiple if statements we can write code which chooses between multiple alternatives.
  • The random() command gives us random numbers which we can use to program (intentionally!) unpredictable behaviours (useful for writing games, for example).
